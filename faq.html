Graph library for Clojure

Loom Philosophy

I just came up with a brilliant way to represent graphs. Can I rewrite Loom to describe graphs the new way?

Designing graph APIs is an entertaining mental activity. Often, when designing an API, once you start using it you'll find many ways to make it simpler/better. Loom has been a very general graph library that contains minimal information about the structure of graphs and is able to run generic graph algorithms on it, so we should not introduce major API changes until the need arises.

What is the relationship of Loom and Titanium? Is there really a need for both?

Titanium is a Clojure API for graph databases that allows for different backends. It provides basic functionality of retrieving elements of the graphs. Loom is a graph algorithms and visualization library. So essentially they are trying to solve different problems: Titanium stores the data, Loom processes the data. Titanium-Loom integration library allows one to run Loom's graph algorithms on the data they have stored in their Titanium and visualize it.

There are lots of graph formats and data structures that one might want to convert to/from Loom. Why doesn't Loom support any of them?

It's important to let the user define their own way of converting their graph to Loom by using Loom's protocols and factory functions. There are infinite number of different graph formats out there, and the primary idea behind Loom is to provide graph algorithms and visualization of the graph, once a graph has been converted into Loom representation. In addition, depending on the problem domain, only the owner of the graph might know how exactly to convert their graph into Loom. This is why Titanium-Loom is implemented as an optional dependency.

Adding Features

I'm interested in adding [an awesome feature and/or graph algorithm] to Loom. How can I do that?

I'd be happy to accept your pull request! Have you signed the Clojure Contributor Agreement (CA)?

Why do I need to sign the CA in order to have my pull request accepted?

The vision for Loom has been to make it a contrib library when it matures, and in order for us to do that, every contributor needs to sign the CA. Therefore, I'll accept pull requests only from contributors who signed the CA. Currently, every code contributor (there's 5 of us so far) has a signed CA.

API Specific Questions

What is the reason for specifying edges sometimes as a vector and other times as two separate arguments?

Loom was created with a convention that one defines an edge by passing in the 2 nodes that define it. The reason you see representation of an edge as a vector in the add-attr-to-edges is that we need a way to represent an edge in a collection of edges over which the function reduces. Destructuring is a more expensive operation and therefore is avoided in the protocol functions.

What is the reason for weight not being stored as attributes?

This allows us to have weights derived from attributes. If one wants to store weights as attributes, one could define a weight function that calls get on the corresponding attribute.
